03

How I build for production

A demo runs once, in front of you. An unattended bot runs at 2 a.m. with nobody watching. These are the habits that separate the two.

The automation is the easy 80%. The boring 20% — exceptions, logging, validation, re-runnability — is what keeps it alive.

None of the builds above are clever. What makes them hold up in production is the engineering wrapped around the clever-looking part: knowing what the bot should do when a row doesn't match, when SAP returns an error, when the file is half-empty, when someone re-runs it twice by accident.

Structure

REFramework & clean state

A consistent skeleton — init, process, end — so any maintainer can read it, and a clean separation between getting work and doing work.

Config

Assets, not hard-coding

Credentials, paths and tunables live in Orchestrator Assets and config, never baked into the workflow. Changing an environment is a setting, not a rebuild.

Exceptions

Business vs system

A flagged exception is handled and logged; an unexpected failure stops loudly. The two are never confused, so a known edge case never masquerades as a crash.

Trust

Validate against source of truth

Every posted record is checked against what the target system actually returned — and logged — so "it ran" and "it worked" aren't assumed to be the same thing.
